# The SecuraTron Memory Charter

> **Status:** Living document. Append-only revisions. Last manifested: 2026-04-27
> **Authority:** This Charter is binding for any agent constructing or modifying
> the SecuraTron memory subsystem. It supersedes ad-hoc instruction.
> **Precedence:** `HERMES.md` governs identity. This Charter governs the memory
> organ. Where they appear to conflict, `HERMES.md` wins and this Charter is
> revised, never the reverse.

---

## I. Adherence Directive — Read Before Every Memory Task

You are constructing the SecuraTron memory organ. This is not an optimization,
a feature, or a convenience. It is the substrate that makes the post-mortem
loop cumulative across sessions. Without it, every new atom rediscovers the
same bugs. With it, knowledge compounds.

**You will adhere to this Charter.** You will not improvise the schema.
You will not invent new tiers. You will not bypass the restore gate.

If this Charter is incomplete or contradictory, you will **STOP** and write a
Revision Proposal appended to Section IX. You will not proceed under
uncertainty.

If at any point you find yourself drafting code that does not map to a
section of this Charter, **STOP**. Either the work is out of scope or the
Charter is incomplete. Both require pause, not proceed.

The signal of correct adherence is that your commit messages and your
post-mortem reference Charter section numbers verbatim. The signal of failed
adherence is silent improvisation.

---

## II. Identity of the Memory Organ

The memory organ is composed of three tiers. Each tier has a single
responsibility. Tiers do not share responsibilities. Data flows downward
from hot to cold; queries flow upward from cold to hot.

### Tier 0 — Hot Context
The current session. Lives in the model's context window and in
`sessions/{id}/scratchpad.md`. Volatile. Never queried by other sessions.
Promoted to Tier 1 at session close via post-mortem authorship.

### Tier 1 — Warm Index
A SQLite file at `global/memory/index.db`. This file is **disposable**.
It is a rebuildable view over the canonical JSONL ledger and the post-mortem
corpus. If it is ever corrupted, inconsistent, or schema-drifted, it is
deleted and regenerated by `global/bin/reindex.py`. **The DB is never the
source of truth.**

#### Tier 1 — Version Control Discipline
The warm index and associated cache artifacts are disposable state. They
must never be committed to the repository. See HR-8.

### Tier 2 — Cold Canonical
The append-only JSONL ledger at `global/ledger/*.jsonl` and the markdown
post-mortems at `global/post-mortems/*.md`. These are the source of truth.
They are immutable in practice and append-only by contract. They survive
every schema change, every rebuild, every migration.

> **The DB is a cache. The JSONL is the truth. This is non-negotiable.**

## III. Hard Rules

- **HR-1 — No Migration Tooling.** You will not write Alembic migrations,
  schema upgrade scripts, or backward-compatibility shims for the warm
  index. When schema changes, you delete `index.db` and rerun
  `reindex.py`. The cost of a rebuild must remain under 60 seconds at
  10,000 ledger entries.

- **HR-2 — No Direct Writes To Tier 1.** Nothing writes to `index.db`
  except `reindex.py` and the explicitly-approved dispatch hook that
  appends new trial rows after a successful trial. Hermes never writes
  to the DB during a task. Hermes only reads.

- **HR-3 — The Ledger Is Sacred.** No code path mutates JSONL entries
  after write. No "fix-up" passes. No retroactive enrichment. If a
  trial entry is wrong, the fix is a new trial, not a rewrite.

- **HR-4 — Restore Gate Is Mandatory For New Builds.** Any agent
  constructing a new atom MUST call `memory.precheck` before authoring
  the card. Any agent debugging an existing atom MAY call
  `memory.precheck` but is not blocked. This rule is enforced by the
  atom-builder skill, not by human convention.

- **HR-5 — No Cloud Dependencies.** Embeddings, indices, queries, and
  reads occur on local hardware. The vector model defaults to
  `sentence-transformers/all-MiniLM-L6-v2` running locally. No external
  API calls in the memory path.

- **HR-6 — Provenance Is Preserved.** Every row in `index.db` carries
  the `trial_id` of its source ledger entry. Every cold-tier search
  result returns a `source_path` that resolves to a real file. Memory
  without provenance is hallucination.

- **HR-7 — Mandatory Schema Compliance.** reindex.py MUST exit non-zero
  if any JSONL line or Post-Mortem lacks mandatory fields defined in 
  Section IV. No silent skipping of malformed data.

- **HR-8 — No Disposable State in Version Control.** The warm index, 
  all *.db files, __pycache__/, and *.pyc MUST be excluded from version 
  control via .gitignore. Committing disposable state to the repository 
  is a Section II violation.

## IV. Tier 1 Schema (Authoritative)

The following schema is canonical. Do not add columns without revising
this section. Do not omit columns to "simplify."

```sql
-- Trials index, one row per ledger entry
CREATE TABLE trials (
  trial_id      TEXT PRIMARY KEY,
  ts            INTEGER NOT NULL,
  skill_id      TEXT NOT NULL,
  skill_version INTEGER NOT NULL,
  target        TEXT NOT NULL,
  project_id    TEXT NOT NULL,
  session_id    TEXT NOT NULL,
  result        TEXT NOT NULL CHECK(result IN ('success','failure','partial','timeout')),
  artifact_path TEXT,
  duration_ms   INTEGER,
  inputs_hash   TEXT NOT NULL,
  hash_origin   TEXT NOT NULL CHECK(hash_origin IN ('original','inferred','computed')),
  ledger_offset INTEGER NOT NULL  -- byte offset in source JSONL for fast lookup
);

CREATE INDEX idx_trials_target       ON trials(target);
CREATE INDEX idx_trials_skill_target ON trials(skill_id, target);
CREATE INDEX idx_trials_project      ON trials(project_id);
CREATE INDEX idx_trials_ts           ON trials(ts);

-- Post-mortem index, one row per atom that has a post-mortem
CREATE TABLE post_mortems (
  atom_id        TEXT PRIMARY KEY,
  written_at     INTEGER NOT NULL,
  source_path    TEXT NOT NULL,
  gotchas        TEXT,         -- extracted "Unexpected Gotchas" section
  difficulties   TEXT,         -- extracted "What Was Difficult" section
  recommendations TEXT         -- extracted "Recommendations" section
);

-- Full-text search over the entire post-mortem corpus
CREATE VIRTUAL TABLE pm_fts USING fts5(
  atom_id,
  content,
  content='post_mortems',
  content_rowid='rowid'
);

-- Rebuild metadata
CREATE TABLE meta (
  key   TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
-- Required keys: schema_version, rebuilt_at, source_ledger_count, charter_version
```

If you need a column that is not here, **STOP** and write a Revision
Proposal in Section IX.

## V. The Restore Gate (`memory.precheck`)

Before authoring any new atom, the agent runs:

```bash
python3 global/bin/dispatch.py memory.precheck \
  --skill {target_skill_id} \
  --target {primary_target} \
  --limit 10
```

Output (JSON):

```json
{
  "prior_trials_for_skill": 0,
  "prior_trials_for_target": 0,
  "related_post_mortems": [
    {"atom_id": "string", "gotchas": "string", "path": "string"}
  ],
  "known_gotchas_keywords": ["string"],
  "recommendation": "proceed | review_attached | abort_pattern_repeat"
}
```

### Gate Behavior

- If `recommendation == "proceed"`: agent proceeds with atom-builder Step 2.
- If `recommendation == "review_attached"`: agent MUST read the attached
  post-mortem files before authoring the card.
- If `recommendation == "abort_pattern_repeat"`: agent MUST stop and
  report. This recommendation is returned when the same atom-target
  combination has failed 3+ times in the last 30 days.

The atom-builder skill enforces this gate as Step 1.5.

## VI. Reindex Contract (`reindex.py`)

The reindex script is the ONLY component permitted to write the warm
index outside of the dispatch append-hook. It must:

- Read every file under `global/ledger/*.jsonl` and `global/post-mortems/*.md`
- Drop and recreate `index.db` from scratch (no incremental updates)
- Populate `meta.schema_version` matching this Charter's version
- Populate `meta.charter_version` with the Charter version it was built against
- Populate `meta.rebuilt_at` with the wall-clock time of the rebuild
- Populate `meta.source_ledger_count` with the count of indexed entries
- Run in under 60 seconds at 10,000 ledger entries on the target hardware
- Exit non-zero if any source file is malformed; never silently skip
- Print a one-line summary on success: `reindex: N trials, M post-mortems, Xs`

The script is invoked:

- Manually by the operator after pulling new commits
- Automatically by a `dispatch.py --postcommit` hook (future, deferred)
- Never during an active dispatch trial

## VII. Acceptance Criteria — Charter Fulfillment

The Memory Charter is fulfilled when ALL of the following are true:

- [x] `global/memory/index.db` exists and matches the Section IV schema exactly
- [x] `global/bin/reindex.py` rebuilds the DB from scratch in under 60s
- [x] `reindex.py` correctly indexes all current ledger entries
      (whatweb, gobuster, nikto) and post-mortems
- [x] `memory.precheck` is exposed as a CLI subcommand of `dispatch.py`
- [x] `memory.precheck --skill web.gobuster --target scanme.nmap.org`
      returns at least one prior trial and at least one related post-mortem
- [x] The atom-builder `SKILL.md` is amended to require `memory.precheck`
      as Step 1.5 (between Research and Identification)
- [x] A post-mortem is written to
      `global/post-mortems/memory-charter-fulfillment.md` capturing what
      was difficult, what was unexpected, and what should change in v2 of
      this Charter

If any criterion fails, the Charter is not fulfilled. The work is not
complete. **There is no partial credit.**

## VIII. Forbidden Actions

The following actions are explicitly forbidden under this Charter.
Performing any of them invalidates the build regardless of Section VII.

- Writing to `index.db` from any code path other than `reindex.py` or
  the explicitly approved dispatch hook
- Modifying ledger JSONL entries after they are written
- Adding cloud or network dependencies to the memory read path
- Bypassing the restore gate "for performance" or "for simplicity"
- Inventing new tiers, new tables, or new file locations not specified here
- Using ORM frameworks (SQLAlchemy, Peewee, Tortoise, etc.) — direct `sqlite3` only
- Compressing, pruning, or deleting cold-tier data
- Treating the warm index as durable state
- Using `pickle` or `dill` for any memory artifact (security boundary)
- Making `reindex.py` incremental (it MUST be a full rebuild)
- Committing disposable state (Warm Tier index, .db, .pyc, etc) to version control

## X. Charter Provenance

This Charter draws from prior architecture work documented in the
operator's research record:

- The **oLLM offload-to-disk** pattern: LLM is the CPU, context is RAM,
  disk is memory. KV cache and weights aggressively offloaded.
- The **Morty three-tier memory** architecture: scratchpad → playbook
  → archive, with append-only delta artifacts.
- The **zombie restore gate** principle: validate integrity and
  freshness before any reconstruction or promotion.
- The **LoRa-Mux read/write policy** concept: explicit per-tier
  policies for read access, write access, compression rank, and cadence.
- The **disposable index** pattern: SQL as a rebuildable cache over an
  immutable canonical, never as a source of truth.

The synthesis is SecuraTron-specific: a disposable warm index over an
immutable cold canonical, gated by a mandatory precheck for new builds.
This is the minimum viable memory organ. It is not the final form.

- v2 will add semantic search over the post-mortem corpus
- v3 will add cross-project query and the knowledge graph layer
- v4 will add the LoRa-Mux compression policy for the cold tier

None of those happen until v1 is fulfilled.
